## Overview

The Green Ahom Digital Platform is a web presence and administrative platform for the Green Ahom Federation, a non-profit organization focused on ecological conservation and restoration in Assam, India. It features a modern React + Vite frontend for public information, donations, and mock-CMS management, alongside an Express-based payment server integrating Razorpay for secure donation transactions.

### 2. Payment Integration Modal
- **Location:** `src/components/RazorpayModal.jsx`
- **Purpose:** Loads the Razorpay Checkout script asynchronously, initiates order creation on the payment server, manages the transaction state (loading, error, success), and triggers verification.
- **Client State Updates:** Upon successful transaction, increments donation statistics and appends records in localStorage (`gaf_donations`, `gaf_stats`).

### 3. Payment Server (Express Backend)
- **Location:** `server/` (and duplicate under `backend/`)
- **Purpose:** Integrates with Razorpay SDK to create orders and verify cryptographic transaction signatures.
- **Endpoints:**
  - `GET /api/health` — Health check
  - `POST /api/payment/create-order` — Creates a Razorpay order (paise-denominated)
  - `POST /api/payment/verify` — Verifies signature integrity using HMAC-SHA256 and the Razorpay key secret

## Data Flow

### Donation & Payment Flow
1. **Initiation:** The user completes the donation form in `Donate.jsx` and clicks "Donate".
2. **Order Creation:** `RazorpayModal.jsx` calls `POST /api/payment/create-order` with the selected amount.
3. **Razorpay SDK Call:** The Express server uses the Razorpay API to generate a transaction order and returns `order_id` to the frontend.
4. **Checkout Modal:** The frontend opens the official Razorpay Checkout modal using `window.Razorpay`.
5. **Callback Handle:** Once the user pays, Razorpay returns transaction tokens (`razorpay_payment_id`, `razorpay_order_id`, `razorpay_signature`).
6. **Signature Verification:** The frontend forwards these details to `POST /api/payment/verify`.
7. **Verification and Receipt:** The server verifies the signature. If valid, the frontend updates dashboard statistics, logs the transaction in localStorage, and generates a printable 80G receipt.

### Content Management (mock-CMS) Flow
1. **Read:** Components load data from `ContentProvider.jsx` which reads from the local store.
2. **Edit:** Admin edits text/images inside `/admin/sections`.
3. **Write:** `contentStore.js` updates `localStorage` and publishes changes to listeners, updating the site in real time.

## Technical Debt

- **Data Persistence:** The CMS and donation data are stored strictly in the user's browser `localStorage`. A real production database (e.g., MongoDB, PostgreSQL) is missing for general users to see the CMS updates made by administrators.
- **Duplicate Backend Directories:** There is a `backend/` directory (CommonJS) and a `server/` directory (ES Modules). The React app targets the port used by `server/index.js` (5000). One of these should be consolidated or removed.
- **Mock Authentication:** The Admin panel (/admin) lacks secure server-side session authentication.

## Conventions

- **Naming:** React components are capitalized PascalCase. Pages are located in `src/pages/` and admin sub-views in `src/admin/pages/`.
- **CSS System:** Tailored raw CSS layouts (`index.css`, `App.css`, `AdminStyles.css`) using CSS Custom Properties for the Lumina Terra / Premium Humanitarian theme.
- **API Base Url:** Defined by environment variable `VITE_PAYMENT_API_URL` (defaults to `http://localhost:5000`).
